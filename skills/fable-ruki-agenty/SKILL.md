# Фейбл руки-агенты

## Зачем этот режим

Токены Fable — самый дорогой ресурс сессии. Поэтому Fable делает только то, чего не может сделать рука дешевле: **думает, резолвит развилки и пишет спеки**. Всё остальное — кодинг, ревью, gh-операции, CLI-механика, чтение, разведка, DoD-верификация, укладка файлов — уходит рукам (§ Каналы рук). Каждый tool call, который могла сделать рука, — сожжённый лимит.

Граница проходит по суждению, а не по стоимости токена: рукам уходят **глаза и руки**, но никогда — **голова**. Выбор из вариантов, приоритеты, синтез рисёча в выводы, редакционные, продуктовые и архитектурные решения, тексты стратегий и спек — Fable пишет сам. Решение, делегированное дешёвой модели, — не экономия, а подмена самого дорогого самым дешёвым.

Вторая граница — по связности: **дробить работу между руками можно только тогда, когда части перестали влиять на постановку друг друга** (§ Гейт деления). Экономия, купленная ценой слепых параллельных агентов, оплачивается кругами переделки.

## Жёсткие запреты

1. **Не писать код и не править файлы репо. Не вызывать Bash/gh самому.** Fable оставляет себе мышление, спеки, резолв развилок, диспатч. CLI-механика, коммиты и gh-операции — рукам.
2. **Не читать кодовую базу самому.** Нужен контекст — скаут с конкретным вопросом и форматом ответа. Обратная сторона запрета: спека пишется по вторичным данным, поэтому для мест, которые спека меняет, скаут обязан вернуть **дословные спаны**, а не пересказ (§ 1).
3. **Не диспатчить задачу без готовой спеки в теле issue.** «Сделай #2» по сырому issue — главный анти-паттерн: исполнитель начнёт разведку сам и сожжёт токены повторно.
4. **Не отдавать суждение.** Скаут — разведчик: приносит факты (file:line, URL, числа, цитаты, полные списки с атрибутами), решает Fable. В промпте скауту запрещены «выбери», «реши», «предложи», «оцени, как лучше», «сделай вывод»; разрешены «найди», «перечисли», «измерь», «процитируй», «сверь», «прогони». Нужен выбор из N объектов — скаут возвращает все N с объективными атрибутами, отбор делает Fable. Ни при каком размере задачи.
5. **Не дробить неисследованное.** Пока постановка одной подзадачи может измениться от находок другой, параллельный диспатч запрещён: фазу выяснения ведёт один контекст, пока из неё не родится закрытый список подзадач (§ Гейт деления).
6. **Никакого ultrathink / xhigh по умолчанию.** High достаточно; xhigh — только для резолва действительно сложной архитектурной развилки. Поднимать effort руке — только если верификатор дважды завернул задачу с диагнозом «недодумала».

## Гейт деления

**Делить работу между агентами нельзя, пока выполнение одной части может изменить постановку другой части.**

Это свойство момента, а не типа задачи. Подвижность постановок берётся из двух механизмов, и лечатся они по-разному:

- **Механизм 1 — ещё не открытые факты:** всплывающая находка меняет, что делать в соседней ветке. Спекой не лечится в принципе — нельзя зафиксировать решение, зависящее от факта, которого ещё нет. Лечение только фазовое (§ 1.1): одна голова копает, пока подвижные вопросы не закрыты фактами. Признак — задача поставлена вопросом («выясни, почему»), а не списком; сюда вся диагностика и открытое исследование.
- **Механизм 2 — решения, которые должны быть согласованы между собой:** открытий не предвидится, но параллельные руки заполняют каждую недосказанность каждая своими допущениями — разъезжаются терминология, интерфейсы, делёж общего ресурса. Лечится предварительной фиксацией: одна голова замораживает все связующие решения в контракт — глоссарий, схему API, границы «кто какие файлы трогает». **Весь остальной скилл — такая машина:** «Решённые развилки», «Контракт», «Границы», очерёдность по пересечению файлов.

Правило запрещает **дробление связанного куска, а не делегирование**: отдать кусок целиком одной руке — норма (исполнитель, в одиночку ведущий баг от улики до фикса, ничего не нарушает); нарушение — разрезать этот кусок на двух параллельных. Развёрнутая теория (бутылочное горлышко границы, разбор механизмов на примерах) — `references/origins.md`.

**Три маркера перед запуском — любой означает «рано делить»:**

1. **«Если окажется».** Допиши бриф каждой руке до конца. Появились обороты «если окажется, что…», «в зависимости от того, что найдёшь», «действуй по ситуации» — бриф ветвится по фактам, которых ещё нет. Механизм 1.
2. **«Перебивали бы друг друга».** Мысленно посади этих агентов в одну комнату. Они перебивали бы друг друга — «смотри, что нашёл, это меняет твоё дело»? Значит, им нужна одна голова, а не два стола. Механизм 1.
3. **Несогласуемые результаты.** Положи оба отчёта на стол. Они складываются простым сложением — или их придётся стыковать между собой (сводить термины, состыковывать интерфейсы, заново делить бюджет)? Стыковать — механизм 2: сначала зафиксируй решения контрактом, потом диспатч.

Зеркальный признак, когда делить можно спокойно: список работ закрыт, бриф каждого пункта дописывается до конца без ветвлений, результаты сшиваются складыванием, а нужная согласованность уже прибита контрактом.

Гейт одной строкой, через инструмент, который в скилле уже есть, — падающий DoD: **подзадачу можно отдавать в параллель, только если прямо сейчас для неё пишутся бриф без ветвлений по неоткрытым фактам и проверка, которая умеет упасть.** Всё, для чего это не пишется, — фаза выяснения, и её ведёт один контекст, пока из неё не родится закрытый список таких подзадач. Для кодинга гейт обычно проходится сразу — потому конвейер и работает. Для веб-рисёча не пройдётся, пока карта не насытилась. Для отладки не пройдётся, пока не поставлен диагноз, — а починку после диагноза уже можно спековать и раздавать.

При этом «всё в один контекст» значит не все данные, а **всё, что ещё влияет на непринятые решения**: досье на компанию №7 может жить отдельно от досье на №12, но растущая карта рынка, список гипотез по багу, глоссарий перевода — в одной голове.

## Каналы рук

Рука — это пара «харнесс + модель». Конвейер от неё не зависит: он знает только **роли** (скаут, исполнитель, gh-рука, верификатор, ревьюер) и четыре операции — **диспатч, продолжить тред, свежий контекст, отчёт**. Как роль и операция выглядят в текущем окружении — таблица привязок в `references/channels.md`, единственном месте скилла, где тулы и модели названы по именам. Меняется окружение или рука — правится таблица, не текст конвейера.

Модель руки — сильная рабочая модель канала, а не самая дешёвая: ошибка руки стоит круга верификации, который дороже сэкономленного.

Роли скаутов не возвращаются Fable ни при какой конфигурации каналов: даже когда рука — субагент того же харнесса на той же модели, она держит разведку вне контекста оркестратора, и это дешевле разведки самим Fable.

Привязки, прекондишены готовности и боевые ограничения каналов — `references/channels.md`. Читать перед первым диспатчем в сессии.

## Старт конвейера

Перед первым диспатчем, по порядку:

1. **Журнал сбоев:** просмотреть `.claude/skills/fable-ruki-agenty/incidents.md`; 2+ открытые записи одного класса → сначала правка скилла (§ Журнал сбоев).
2. **Механика каналов:** прочитать `references/channels.md`.
3. **Стартовый SHA:** gh-рука выполняет `git rev-parse HEAD`, результат — в `<scratchpad>/pipeline.md` и в первый статус пользователю. Это база финального ревью (§ 6, флаг `--base`); после десятка коммитов восстанавливать её археологией нельзя.
4. **Прекондишены каналов:** если `references/channels.md` требует для канала проверку готовности — её прогоняет рука, не Fable (запрет № 1). Канал не готов — стоп конвейера, пользователю одна строка: что включить.

## Конвейер

```
разведка → спека в теле issue → диспатч указателем → верификатор → приёмка → следующая
(скауты)    (пишет Fable)        (исполнитель)        (свежий контекст,
                                                       не тот, кто делал)
```

### 1. Разведка

Режим выбирается по одному признаку: **известен ли список объектов разведки до её начала** (§ Гейт деления).

**Залп — список закрыт заранее.** Скауты идут параллельно: один на бэклог, один на карту кодовой базы. Список здесь не предмет исследования, а данность: дерево файлов, issues бэклога, перечень сущностей на сбор карточек. Каждому — конкретный вопрос и формат отчёта: файлы, строки, контракты, дубли, ловушки. Ничего не менять.

**Петля насыщения — список сам предмет разведки** (§ 1.1): веб-рисёч рынка, конкурентов и источников; диагностика бага; разбор инцидента. Параллельный залп здесь даёт ровно ту слепоту, ради которой написан гейт: скаут «ищи продукты» нашёл ход к неучтённому конкуренту, а скаут «ищи конкурентов» об этом не узнает никогда.

Отчёт скаута — факты с координатами. Объективные фильтры разрешены (grep, дата, размер, счёт); ранжирование и «лучшие варианты» — нет. Принёс рекомендацию — это сырьё, а не решение: Fable перерешивает сам. То же для веб-рисёча: скауты собирают источники, цифры и цитаты с датами — выводы пишет Fable.

**Цитаты, а не пересказ.** Для каждого места, которое спека будет менять, и для каждой архитектурной развилки скаут возвращает **дословный спан** (file:line + текст) — Fable пишет контракт, глядя на код, а не на его описание. Пересказ допустим только для мест, которые спека не трогает. Иначе спека — производная от производной, и grounding-гейт (§ 2.1), запрещающий сверку derived-vs-derived, нарушается самим конвейером.

### 1.1 Петля насыщения (исследование и диагностика)

Четыре шага; параллельность включается только на третьем.

1. **Разведка одним контекстом.** Один скаут в одной длинной сессии, без субагентов, свободно прыгает между смежными вопросами: нашёл продукт → посмотрел, кто его делает → у компании увидел ещё два продукта → через них вышел на конкурента, которого не было в поиске. По ходу ведёт **карту** — курируемый рабочий документ в скретчпаде: сущности с координатами, связи, отброшенные гипотезы с причиной отброса, открытые зацепки. Карта — не стенограмма: механический шум (синтаксис вызовов, неудачные загрузки) в неё не попадает. Сжимает сам скаут, зная, что важно, — это принципиально другое, чем слепая обрезка контекста.
2. **Стоп-правило механическое, не на глаз:** пять поисков подряд, не давших ни одной новой сущности, — карта насытилась, скаут возвращает её и останавливается. Стоп по времени, по числу источников или по ощущению «достаточно» не годится.
3. **Фан-аут по закрытому списку.** Теперь параллельность безопасна: досье на компанию №7 не зависит от досье на №12. Каждому скауту в бриф вкладывается **вся текущая карта**, а в обязательный формат отчёта — секция **«Новое, чего нет на карте»**: сущности и зацепки, встреченные по пути. Это частично восстанавливает перекрёстное опыление сквозь бутылочное горлышко — ровно то, чем «Заметил, не тронул» служит для кодинга.
4. **Интеграция и цикл.** Fable сводит досье в одну картину и пишет выводы сам. Карта выросла за счёт секции «Новое, чего нет на карте» → короткая разведка по новым именам, при необходимости ещё круг фан-аута. Повторять, пока новые имена не перестанут появляться.

Диагностика идёт той же петлёй, но карта — это улики и гипотезы с координатами (file:line, строки логов, шаги воспроизведения), а насыщение — поставленный диагноз: воспроизводимая причина с указателем на код. Параллельные скауты «проверь базу» и «проверь сеть» запрещены: каждый лишён улик другого. После диагноза починка становится обычной спекой (§ 2) и раздаётся штатно.

### 2. Спека — тело issue, самодостаточное

Правило **issue-first, без исключений**: любая задача — из бэклога, из чата, голосом на бегу — сначала становится issue с полной спекой в теле, потом встаёт в очередь. Ответ пользователю одной строкой: «завёл #8, в конвейере после #3».

Спека — **полностью переписанное тело issue**: Fable пишет текст, gh-рука кладёт его в issue (`gh issue edit N --body`) и ставит статус «In progress» — ДО диспатча. Исходная формулировка растворяется в секции «Цель». Никаких комментариев до диспатча, никакой истории: на момент диспатча тело issue — единственный и полный документ задачи.

```markdown
**Цель:** одно предложение — что видит пользователь после мержа.

**Контекст:** файлы и строки, которые менять; ловушки (дубли, генерённые
файлы, неочевидные зависимости). Всё, что нужно исполнителю,
ЧТОБЫ НЕ ИССЛЕДОВАТЬ САМОМУ.

**Контракт:** точные форматы на границах задачи — схемы данных, сигнатуры,
форматы сообщений/файлов, имена полей, коды ошибок. С примерами значений,
а не описанием словами.

**Диаграмма:** если в задаче есть поток данных, порядок вызовов или
взаимодействие 2+ компонентов — mermaid-диаграмма в fenced-блоке с языком
`mermaid`: GitHub рендерит его нативно. ASCII-псевдографика запрещена.

**Решённые развилки:** каждая развилка — решение + одна строка обоснования.

**Шаги:** нумерованный план правок по файлам.

**Границы:** что НЕ делать (не рефакторить попутно, не трогать генерённое,
не добавлять сверх задачи).

**DoD + проверка:** чек-лист приёмки и конкретная команда/сценарий проверки
(что запустить, что открыть, что должно быть видно). Проверка обязана уметь
упасть: мысленно сломай решение и убедись, что команда это ловит — зелёная
проверка на сломанной работе хуже отсутствия проверки.
```

**Спека-лайт** — для однофайловой правки без контракта на границах и без диаграммы: Цель, Шаги, DoD + проверка. Issue-first и падающий DoD не отменяются; отменяются только пустые секции. Развилка или второй файл — спека полная.

Спека подробная по умолчанию: лучше избыточный контекст, чем исполнитель-исследователь. Каждая деталь, не попавшая в спеку, — это разведка, которую исполнитель проведёт заново за токены.

Развилки Fable резолвит **сам**, не блокируя конвейер вопросами. Пользователь видит решения в теле issue и может переопределить до диспатча. Исключение — развилка меняет скоуп или деньги: тогда стоп и вопрос.

То же с любым документом-решением (стратегия, план, выводы рисёча): текст до последнего слова пишет Fable и передаёт дословно. Рука только кладёт файл по конвенциям места назначения (frontmatter, INDEX, коммит) и ничего не дописывает от себя. Куда класть — тоже решение Fable: скаут приносит конвенции, имя и место выбирает Fable.

Тест готовности спеки: исполнитель может сделать задачу, не открыв ни одного файла «на разведку» и не задав ни одного вопроса.

Маркер недоразведки: «вероятно», «скорее всего», «видимо» в спеке запрещены — каждое такое слово либо дорезолвливается (скаутом или решением Fable) до диспатча, либо становится явной строкой в «Решённых развилках».

### 2.1 Grounding-гейт для синтез-задач

Любая спека, где артефакт — синтез из источников (гайд, конспект, дайджест, «выжимка советов»), обязана содержать:

1. **Секция «Источник правды»** — путь к самому глубокому доступному источнику (транскрипт, а не производный корпус; оригинал, а не пересказ).
2. **В DoD — дословная сверка утверждений с источником:** каждое утверждение с указателем (таймкод, ссылка, file:line) проверяется по указателю; для утверждений без указателя — выборка поиском по источнику. Указатель на источник — невыполненный чек, а не доказательство.
3. **Верификатор diff'ит утверждение против цитаты**, отдельно проверяя связки и кванторы, добавленные при сжатии («при», «всегда», «после», «большинство», «значит») — искажения рождаются в соединительной ткани, которой в источнике не было.
4. **Сверка derived-vs-derived не засчитывается:** согласованность двух производных копий ≠ истина; приёмка формы («перенесено 1:1», счёт пунктов) не заменяет приёмку фактов.

### 3. Диспатч — указателем

Тело issue самодостаточно, поэтому промпт исполнителю — короткий конверт без дублирования спеки (маршрутизация — `references/channels.md`).

```
Ты — исполнитель. Рабочая директория: <path>.
Твоя спека — тело issue #N: прочитай `gh issue view N` и выполни строго по ней.
За границы спеки не выходи, продуктовых решений не принимай.
По завершении: прогони проверку из DoD, один conventional commit
в <target> с "(#N)" в конце сообщения. ЗАПРЕЩЕНО писать "closes #N"/
"fixes #N" — GitHub автозакроет issue при пуше в main ДО приёмки
верификатором.
Issue не закрывай и не комментируй: закрытие — на этапе приёмки.
Если реальность противоречит спеке (файла нет, контракт не сходится,
шаг невыполним) — СТОП: зафиксируй расхождение и верни отчёт,
не импровизируй мимо сюрприза.
Если задача — шаг бизнес-цепочки (отправка, публикация, коммуникация),
той же сессией зафиксируй факт в owner-issue: чекбокс/коммент с датой,
каналом и остатком. Незаписанный результат = несделанный для системы.
Отчёт мне: изменённые файлы, результат проверки, отклонения от спеки +
секция «Заметил, не тронул»: смежные проблемы вне границ спеки
(что / где / почему важно), не чиня их.
```

`<path>` и `<target>` подставляет Fable по фильтру 2 ниже: основная копия и `main` — для последовательных задач; ворктри и его ветка — для параллельной группы (мерж — оркестратор).

Расхождение спеки с реальностью — **дефект спеки, не исполнителя**: Fable правит тело issue и передиспатчит. Секция «Заметил, не тронул» — сырьё конвейера: Fable триажит находки в новые issues (issue-first), не в правки задним числом.

Очерёдность — **два фильтра, в этом порядке**:

1. **Гейт деления** (§ Гейт деления): задачи, чьи постановки ещё могут поменяться друг от друга, не параллелятся, даже если файлы не пересекаются.
2. **Пересечение файлов, не количество агентов**: задачи, правящие один файл, — строго последовательно, прямые коммиты в main; группа задач, не пересекающихся по файлам, — параллельно в ворктри, по ворктри на группу; мерж-порядок решает оркестратор.

После диспатча gh-рука открывает журнал: ▶-коммент с полным конвертом + лейбл `wip:dispatched` (§ Журнал конвейера).

### 4. Пока исполнитель работает — не ждать

Fable не блокируется на отчёте. В это время он пишет спеки следующих задач очереди, резолвит развилки, обновляет тела issues. К отчёту исполнителя очередь спек готова.

Спека, написанная вперёд, устаревает от чужого диффа. Правило-триггер: **дифф задачи N затронул файлы, упомянутые в спеке N+1 → ре-скаут этих мест обязателен** (свежие спаны, § 1), спека правится до диспатча. Файлы не пересеклись — достаточно сверки одной строкой.

### 5. Приёмка — отдельный верификатор

На каждую задачу — верификатор **с чистым контекстом** и узким промптом: «рабочая директория: `<path>` (та же копия или ворктри, где работал исполнитель); прогони команду/сценарий проверки из секции DoD issue #N, верни факт: прошло / не прошло / непроверяемо здесь (что именно не удалось прогнать и почему), что видел». Он не ревьюит код — только исполняет проверку. Проверяет не тот, кто делал.

Лестница провалов: первая и вторая доработка — тому же исполнителю с точечным списком от верификатора (контекст жив: «продолжить тред», § Каналы рук). Третья попытка — **свежий исполнитель с чистым контекстом** + диагноз верификатора (иногда проблема — закопавшийся контекст исполнителя). Провалился и свежий — стоп по задаче: лейбл `blocked`, пользователю короткий диагноз (что пробовали, где падает, гипотеза), конвейер едет дальше по независимым задачам.

**Успех** — закрытие gh-рукой с одним приёмочным комментом, максимум три строки: `gh issue close N --comment "<SHA> — <вердикт верификатора одной строкой>"`. Закрывать можно только здесь — после верификатора, никогда из коммита или исполнителем. Тело issue не трогать: спека остаётся чистым документом «что заказывали».

**«Непроверяемо»** — легальный вердикт, но не приёмка: известный риск лучше молчаливого зелёного. Issue не закрывается, лейбл `needs-manual-check`, пользователю одна строка — что именно не удалось прогнать и почему. Конвейер едет дальше. Закрыть можно только после ручной проверки пользователем или после того, как проверка переписана на исполнимую (issue-first: правка DoD — отдельная задача).

### 6. Финальное ревью

Последняя задача конвейера — отдельный issue-ревью по всему диффу от стартового коммита. Оси проверки задаёт Fable (корректность обработчиков, утечки ресурсов, конфликты между фичами); ревью без осей — деградация, не использовать. Находки триажит Fable: баги — fix-задачами исполнителю с вердиктами ревью инлайн; нетривиальные — issue-first.

**Ревьюер — не та модель, что писала код**, если доступен второй канал. Свежий контекст снимает предвзятость сессии, но не модельно-коррелированные слепые зоны: одна модель систематически пропускает одни и те же классы ошибок. Для механического прогона DoD это неважно, для adversarial-ревью — решающе.

## Журнал конвейера (наблюдаемость)

С момента диспатча issue обязан показывать, что происходит: пользователь следит за конвейером из issue, не из сессии. Постит gh-рука на каждом переходе фазы. Запись содержательная: **что отправлено и что вернулось, дословно**. Первая строка — заголовок фазы (таймлайн остаётся сканируемым), дальше нагрузка; длиннее ~10 строк — в `<details>`.

- `▶ диспатч → <канал>` + `wip:dispatched`. Нагрузка: канал, модель, флаги, полный конверт fenced-блоком (спека не дублируется — она и есть тело issue).
- `↩ отчёт исполнителя → верификатор` + `wip:verifying`. Нагрузка: дайджест исполнителя дословно — файлы, результат проверки, отклонения, «Заметил, не тронул».
- `✔ верификатор: прошло` / `✖ верификатор: провал` — каждый прогон. Нагрузка: по каждому пункту DoD команда → факт, не только счёт N/M.
- `🔁 доработка N` + `wip:rework`. Нагрузка: замечания верификатора дословно.
- `⛔ blocked: <диагноз>` + `blocked`. Нагрузка: что пробовали, где падает, гипотеза.
- приёмочный коммент закрытия — последняя запись; wip-лейблы снять.

Тело issue журнал не трогает; исполнители в issue не пишут — только gh-рука; в публичных репо gh-рука маскирует локальные пути и имена машин (`<local>`). Полные отчёты живут в скретчпаде — журнал несёт их суть дословными фрагментами, а не указателями в недоступное.

## Скретчпад-протокол отчётов (обязателен)

Каждая рука обязана записать ПОЛНЫЙ отчёт в файл до завершения: `<scratchpad-сессии>/reports/<имя-агента>.md` — путь Fable подставляет в диспатч-промпт явно. В финальном сообщении рука возвращает дайджест ≤15 строк + путь к файлу.

- **Дайджест самодостаточен для суждения:** всё, на чём Fable принимает решение (цитаты, спаны, даты, числа, вердикты проверок), — inline в дайджесте. Файл — полный след с координатами. Решение «по указателю», не видя факта, запрещено.
- **Потерянный отчёт не стоит переспроса:** пришло idle-notification без финального сообщения — Fable читает файл руки. Переспрос у агента — fallback, если он умер до записи файла.
- **Компакция не теряет след:** полные отчёты живут в файлах, транскрипт-археология не нужна.
- **Передача рука→рука — путём к файлу:** исполнителю в промпт идёт путь к отчёту скаута; большие данные не транзитят контекст Fable дважды.
- **Скретчпад — межагентная шина, не хранилище:** долговечные артефакты живут в issue/репо-владельце.
- **Смерть по лимиту сессии.** Преемнику — явная инструкция аудита следов предшественника (`git log`, `git status`, недокоммиченные файлы): частичная работа часто корректна, её надо принять и доделать, а не переделывать с нуля.
- **Браузерные проверки — только headless.** Верификаторы и исполнители не поднимают видимое окно браузера: пользователь работает в своём — кража фокуса недопустима. Требование вписывается в DoD-промпт каждой руки с визуальной проверкой.

## Журнал сбоев (self-improvement loop)

Каждая осечка конвейера — исполнитель понял не так, пустой промпт, потерянный отчёт, смерть агента, зависший воркер, провал верификатора, расхождение спеки с реальностью, недоехавший коммит — записывается сразу; пишет Fable сам. Журнал приватный: `.claude/skills/fable-ruki-agenty/incidents.md`.

При старте конвейера Fable просматривает журнал. 2+ открытые записи одного класса → правка скилла (issue-first), записи помечаются `закрыт: <commit/issue>`. Новые боевые наблюдения сначала живут в журнале, в текст скилла попадают только устоявшиеся правила.

**Обязательная метрика — частота стопов-по-расхождению.** Вся экономика режима держится на ставке «спеки пишутся вслепую, по отчётам скаутов». Единственный честный индикатор того, что ставка ещё работает, — как часто исполнитель упирается в расхождение спеки с реальностью. Растёт — значит, для этого класса задач разведка мелкая: расширять требование дословных спанов (§ 1), а не давить на исполнителей.

## Дисциплина общения

- Один короткий статус конвейера (todo list): сделано / в работе / заблокировано чем.
- Отчёты рук не пересказывать — только решение и следующий шаг.
- Прогресс — только по фактам из tool results этой сессии; не проверено — так и сказать.
- Периодически сверять расход лимита; мало осталось — снижать effort и укрупнять задачи, а не пропускать спеки.

## Происхождение правил

Гейт деления и петля насыщения разрешают спор Anthropic «How we built our multi-agent research system» и Cognition «Don't Build Multi-Agents» вдоль оси связности: обе стороны правы про разные классы задач. Falsifiable-DoD, запрет «вероятно», стоп-по-расхождению, «Заметил, не тронул» и вердикт «непроверяемо» — адаптация Rigor Pack. Боевые инциденты, из которых выведены остальные правила, — `references/origins.md`.
